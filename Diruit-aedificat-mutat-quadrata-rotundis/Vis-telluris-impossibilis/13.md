# Chapitre 13 — Searchlores face aux autres Frameworks IA : une nouvelle catégorie ?

> *« Les bons frameworks répondent à une question. Les grands frameworks changent la question. »*

---

# Une erreur fréquente

Depuis plusieurs années, les comparatifs de frameworks IA ressemblent souvent à ceci :

| Framework | Agents | Tools | RAG | Workflow |
| --------- | ------ | ----- | --- | -------- |

Puis on compte les cases.

Qui en possède le plus.

Qui est le plus rapide.

Qui appelle le plus de LLM.

Ce type de comparaison est pratique...

mais profondément trompeur.

Pourquoi ?

Parce que tous ces frameworks ne cherchent pas à résoudre le même problème.

---

# Avant de comparer...

Posons une question beaucoup plus intéressante.

Quel est le **problème fondamental** que chaque framework tente de résoudre ?

À partir de cette seule question, le paysage devient beaucoup plus clair.

---

# LangChain

L'objectif historique de LangChain est simple.

> Faciliter la construction d'applications utilisant un ou plusieurs LLM.

Son abstraction principale est :

```text
Chain
```

Puis sont apparus :

* les Tools
* les Retrievers
* les Memory
* les Agents

Le cœur reste pourtant identique.

Assembler des briques.

---

# LangGraph

LangGraph apparaît ensuite.

Son intuition est excellente.

Les chaînes deviennent trop rigides.

Les agents ont besoin de revenir en arrière.

De boucler.

De mémoriser.

LangGraph remplace alors la chaîne...

par un graphe.

```text
Node

↓

Edge

↓

State

↓

Node
```

C'est une évolution majeure.

Mais le problème reste :

l'orchestration.

---

# CrewAI

CrewAI pose une autre question.

> Et si plusieurs agents collaboraient ?

Son unité fondamentale devient :

```text
Agent
```

Puis :

```text
Role

↓

Task

↓

Crew
```

La logique principale devient sociale.

Les agents coopèrent.

---

# AutoGen

Chez Microsoft,

AutoGen pousse cette idée encore plus loin.

Les agents dialoguent.

Ils négocient.

Ils se répartissent le travail.

Le raisonnement est distribué.

---

# Haystack

Haystack vient d'un autre univers.

Son problème est :

la recherche documentaire.

Ses abstractions principales sont :

* Document
* Retriever
* Pipeline
* Reader

Nous sommes ici dans le monde du RAG.

---

# LlamaIndex

LlamaIndex poursuit une idée proche.

Son obsession est :

comment organiser efficacement

la connaissance externe.

L'unité fondamentale devient :

```text
Index
```

Tout tourne autour de l'indexation.

---

# Searchlores

Et maintenant...

Searchlores.

Quelle est son abstraction principale ?

Ni un Agent.

Ni un Tool.

Ni un Prompt.

Ni un Pipeline.

Ni un Document.

Ni un Index.

C'est...

```text
Investigation
```

À ce moment précis,

on comprend pourquoi il paraît si différent.

---

# Le véritable changement

Chaque framework considère un objet différent comme fondamental.

LangChain pense :

pipeline.

CrewAI pense :

collaboration.

Haystack pense :

documents.

Searchlores pense :

recherche.

Pas recherche Web.

Recherche scientifique.

# Ce que Searchlores ne cherche pas à faire

Cette partie est importante.

Searchlores n'essaie pas d'être :

* le meilleur chatbot ;
* le meilleur framework RAG ;
* le meilleur orchestrateur d'agents ;
* le meilleur wrapper OpenAI.

Et c'est probablement une bonne chose.

Il reste concentré sur son domaine.

---

# Là où il devient unique

En parcourant tous ces frameworks,

j'ai progressivement compris où Searchlores change réellement les règles.

Il formalise explicitement :

* les hypothèses ;
* les preuves ;
* les contradictions ;
* les récits ;
* les connaissances ;
* leur évolution.

À ma connaissance,

aucun autre framework majeur ne place ces objets au centre de son architecture.

---

# Le rapport au temps

Autre différence fascinante.

Chez beaucoup de frameworks,

chaque exécution est indépendante.

```text
Question A

↓

Réponse
```

Puis :

```text
Question B

↓

Réponse
```

Searchlores préfère ceci.

```text
Question A

↓

Lore

↓

Question B

↓

Lore enrichi

↓

Question C

↓

Lore enrichi
```

Le temps devient une dimension du système.

# Le rapport à la mémoire

LangChain possède une mémoire.

CrewAI possède une mémoire.

LangGraph possède un état.

Searchlores possède...

une histoire.

Ce n'est pas exactement la même chose.

La mémoire conserve.

Le Lore évolue.

# Le rapport aux preuves

Voici probablement la plus grande différence.

Dans un RAG classique :

les documents servent à répondre.

Dans Searchlores :

les preuves servent à raisonner.

Cette nuance change profondément l'architecture.

# Peut-on les combiner ?

Absolument.

Et c'est probablement l'avenir.

Imaginons.

LangGraph orchestre.

Haystack recherche les documents.

Searchlores construit le Lore.

Searchlores pourrait devenir...

Cette idée me paraît beaucoup plus intéressante que celle d'un remplacement.

# Les limites de Searchlores

Pour être juste,

il faut aussi parler de ses limites.

Aujourd'hui,

si tu souhaites simplement :

* créer un chatbot ;
* résumer un document ;
* appeler GPT ;
* construire un assistant.

Searchlores est probablement trop ambitieux.

Son modèle métier demande un investissement intellectuel important.

Il suppose que la connaissance mérite d'être structurée.

Toutes les applications n'ont pas ce besoin.

---

# Les domaines où il pourrait exceller

En revanche,

plus le problème devient complexe,

plus Searchlores paraît pertinent.

Par exemple.

* Veille scientifique

* Recherche académique
* Analyse géopolitique
* Cyber Threat Intelligence

la réponse est moins importante...

que la manière dont elle a été construite.

# Une nouvelle catégorie ?

En terminant cette comparaison,

je me suis posé une question.

Et si Searchlores n'appartenait finalement à aucune des catégories actuelles ?

Ce n'est pas exactement :

* un framework d'agents ;
* un framework RAG ;
* un orchestrateur ;
* un moteur de workflows.

Je serais tenté de le qualifier de :

> **Cognitive Investigation Framework**

Ou peut-être :

> **Knowledge Investigation Platform**

Ces expressions n'existent pas vraiment aujourd'hui.

Et c'est peut-être le signe que Searchlores tente effectivement d'ouvrir un nouvel espace de conception.

---

# Regard critique : une ambition qui dépasse encore l'écosystème

C'est sans doute la principale conclusion de cette comparaison.

Searchlores ne surclasse pas systématiquement les autres frameworks. Dans de nombreux cas d'usage simples, ils seront plus rapides à prendre en main et plus adaptés.

En revanche, Searchlores poursuit une ambition différente : faire de la **construction de connaissances** un objet logiciel à part entière. Cette ambition est suffisamment originale pour qu'il soit parfois difficile de le comparer directement à ses contemporains. Les catégories habituelles — agents, RAG, workflows — ne suffisent plus à le décrire.

Reste une question importante : cette vision séduira-t-elle une communauté suffisamment large pour donner naissance à un véritable écosystème ? L'histoire des frameworks montre que les meilleures idées ne s'imposent pas seulement par leur élégance technique, mais aussi par leur capacité à être comprises, adoptées et prolongées par d'autres développeurs.

---

# Une réflexion personnelle

En refermant ce chapitre,

je me suis souvenu d'une phrase souvent attribuée à Alan Kay :

> *« Le meilleur moyen de prédire le futur est de l'inventer. »*

Je ne sais pas si Searchlores deviendra un framework majeur.

Personne ne peut le savoir.

En revanche,

je suis convaincu d'une chose.

Il propose une manière différente de penser les systèmes d'IA.

Et cette différence,

à elle seule,

méritait que nous consacrions treize chapitres à l'explorer.

---

# Conclusion

Nous avons désormais replacé Searchlores dans le paysage des frameworks IA contemporains. Cette comparaison montre qu'il ne cherche pas à être un concurrent direct de LangChain, LangGraph ou CrewAI, mais qu'il s'inscrit dans une autre logique : celle de l'investigation, de la mémoire cumulative et de la structuration durable de la connaissance.

Le **chapitre 14** constituera l'aboutissement technique de cette monographie. Nous y quitterons les comparaisons pour porter un regard d'architecte sur le projet lui-même. Nous identifierons ses points forts, ses fragilités, les défis qui l'attendent, les pistes d'évolution les plus prometteuses et les décisions qui pourraient conditionner son avenir. Ce sera une véritable **revue d'architecture**, comme on en réalise avant l'adoption d'un framework stratégique au sein d'une entreprise. À mes yeux, ce sera probablement le chapitre le plus exigeant… et aussi le plus utile pour un décideur technique.
