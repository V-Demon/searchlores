# Chapitre 7 — Le DSL *Lore* : écrire la connaissance plutôt que le code

> *« Un bon DSL ne réduit pas la complexité. Il la rend intelligible. »*

---

# Pourquoi inventer un langage ?

Lorsqu'un framework gagne en maturité, une question finit toujours par apparaître :

> *Comment représenter le domaine sans écrire du code ?*

Au début, quelques classes Python suffisent.

Puis viennent les fichiers JSON.

Ensuite les YAML.

Enfin…

on découvre qu'aucun de ces formats n'exprime réellement le domaine.

C'est exactement ce qui semble avoir conduit Searchlores vers le DSL **Lore**.

---

# Un constat simple

Prenons une enquête classique.

On pourrait la décrire en Python.

```python
investigation = Investigation(
    topic="Prompt Injection",
    hypotheses=[...],
    evidence=[...],
    reports=[...]
)
```

Cela fonctionne.

Mais ce code mélange plusieurs niveaux :

* la logique métier ;
* la syntaxe Python ;
* l'instanciation des objets.

Or une enquête n'est pas un programme.

C'est une connaissance.

---

# La naissance d'un DSL

Le DSL (Domain Specific Language) apparaît lorsque les auteurs font un choix très fort.

Ils décident que l'objet principal du framework n'est plus une classe Python.

C'est un document.

Autrement dit :

```text
Lore

↓

Parser

↓

Objets

↓

Investigation
```

Le code devient secondaire.

---

# Un parallèle avec Kubernetes

Cela peut surprendre.

Pourtant, Searchlores suit une évolution très proche de Kubernetes.

Dans Kubernetes :

```yaml
Deployment
```

ne représente pas du code.

Il représente une intention.

Le moteur se charge ensuite de transformer cette intention en actions.

Le DSL Lore poursuit exactement cette idée.

Il décrit :

* une enquête ;
* des concepts ;
* des relations ;
* des hypothèses.

Le moteur décide ensuite comment les exploiter.

---

# Décrire plutôt que programmer

C'est probablement la meilleure manière de comprendre le Lore.

On ne demande plus :

> Exécute ceci.

On décrit :

> Voici ce que je connais.

ou

> Voici ce que je cherche.

La différence est immense.

---

# Une connaissance déclarative

Prenons un exemple imaginaire.

```yaml
topic: Prompt Injection

questions:
  - Pourquoi fonctionne-t-elle ?

hypotheses:
  - Le modèle suit des instructions contradictoires

sources:
  - OWASP

goals:
  - Cartographier les vulnérabilités
```

Ce document n'est pas un script.

C'est une photographie d'un état de connaissance.

---

# Pourquoi est-ce si intéressant ?

Parce qu'un DSL possède des propriétés qu'un programme ne possède pas.

Un document peut être :

* versionné ;
* comparé ;
* fusionné ;
* partagé ;
* relu par un humain ;
* généré automatiquement.

Autrement dit,

la connaissance devient portable.

---

# Une séparation remarquable

L'un des choix les plus élégants du projet est de séparer :

```mermaid
graph LR

Lore --> Parser

Parser --> Investigation

Investigation --> Engine

Engine --> Report
```

Chaque couche possède sa responsabilité.

Le DSL n'a aucune connaissance du moteur.

Le moteur n'a aucune connaissance de la syntaxe.

Cette indépendance facilite énormément les évolutions.

---

# Le DSL comme contrat

Un autre aspect m'a frappé.

Le Lore ressemble davantage à un **contrat** qu'à un fichier de configuration.

Il ne dit pas :

> Comment faire.

Il dit :

> Ce qui existe.

Cette nuance rappelle fortement les architectures déclaratives modernes.

---

# Les objets du langage

En lisant les exemples et les spécifications du dépôt, on voit émerger plusieurs familles de concepts.

## Les sujets

Ils définissent le terrain de l'enquête.

Exemple :

```
Prompt Injection
```

---

## Les questions

Chaque investigation possède plusieurs interrogations.

Toutes n'ont pas forcément une réponse immédiate.

---

## Les hypothèses

Le framework ne considère jamais une hypothèse comme un fait.

Cette distinction est fondamentale.

Une hypothèse est un objet de premier ordre.

---

## Les preuves

Les preuves viennent soutenir ou invalider les hypothèses.

Elles peuvent évoluer au fil de l'enquête.

---

## Les relations

Les concepts ne sont jamais isolés.

Le DSL cherche donc à représenter explicitement leurs liens.

---

# Une syntaxe tournée vers l'humain

Contrairement à beaucoup de DSL industriels,

le Lore reste très lisible.

On retrouve une influence évidente :

* de YAML ;
* des fichiers Markdown ;
* des documents de recherche.

Autrement dit,

le langage cherche moins à être compact qu'à être compréhensible.

---

# Le DSL comme mémoire collective

Imaginons une équipe.

Chaque membre mène une investigation différente.

Dans un framework classique,

les résultats seront souvent des scripts Python.

Dans Searchlores,

ils deviennent des documents Lore.

Ces documents peuvent ensuite être :

* fusionnés ;
* enrichis ;
* comparés ;
* archivés.

On construit progressivement une bibliothèque de connaissances.

---

# Une idée très moderne

Cette approche rejoint un mouvement de fond dans l'ingénierie logicielle.

Nous voyons apparaître de plus en plus de systèmes où :

les données pilotent le moteur.

Plutôt que :

le moteur pilote les données.

Exemples :

* Kubernetes
* Terraform
* OpenAPI
* GitHub Actions

Searchlores applique cette philosophie…

à la connaissance.

---

# Un moteur piloté par des modèles

Autrement dit :

```text
Lore

↓

Validation

↓

Parser

↓

Investigation

↓

Engine

↓

Report
```

Le DSL devient le point d'entrée naturel du système.

Imaginons que tu veuilles créer une nouvelle enquête.

Avec un framework traditionnel :

Avec Searchlores,

Cette différence paraît anodine.

# Une ouverture vers les outils IA

C'est ici que j'entrevois l'un des potentiels les plus intéressants du projet.

Parce que le Lore est déclaratif,

un LLM pourrait lui-même :

* générer un Lore ;
* enrichir un Lore ;
* fusionner plusieurs Lore ;
* corriger un Lore ;
* traduire un Lore.

Autrement dit,

le DSL devient un langage commun

entre les humains

et les intelligences artificielles.

Je pense que cette partie du projet anticipe une évolution importante de l'écosystème IA.

Aujourd'hui,

---

# Regard critique : une intuition brillante, un chantier encore ouvert

C'est le premier chapitre où je prends un peu de recul par rapport au dépôt lui-même.

L'idée d'un DSL est excellente. Elle place la connaissance au-dessus de l'implémentation, favorise le partage, la relecture, le versionnement et ouvre la voie à des outils d'analyse ou de génération automatique.

En revanche, en l'état actuel du projet, le DSL apparaît davantage comme une **grammaire en construction** que comme un langage pleinement stabilisé. On en devine les règles, les abstractions et les usages, mais plusieurs aspects semblent encore évoluer : le vocabulaire, les conventions, les mécanismes de validation et la manière dont certains concepts sont matérialisés dans le code.

Ce n'est pas inhabituel. Beaucoup de DSL célèbres — HCL (Terraform), Docker Compose ou même Kubernetes YAML — ont mis plusieurs années avant d'atteindre leur forme actuelle. Searchlores semble suivre une trajectoire comparable : il pose d'abord les concepts, puis laisse l'usage guider leur stabilisation.

---

# Une intuition personnelle

En refermant ce chapitre, j'ai eu une impression assez forte.

À mon sens, Searchlores ne cherche pas seulement à créer un framework.

Il cherche à créer un **langage de l'investigation**.

Si cette intuition se confirme dans les prochaines évolutions du projet, alors le code Python pourrait progressivement devenir une simple implémentation de référence. Le véritable cœur de Searchlores serait alors le Lore lui-même, comme SQL est devenu plus important que les moteurs de bases de données qui l'exécutent.

C'est une ambition considérable.

Et c'est aussi ce qui rend ce dépôt si singulier.

# Conclusion

Nous avons maintenant exploré les trois piliers conceptuels de Searchlores :

* **l'Investigation**, qui donne un cadre à la recherche ;
* **le Lore**, qui transforme les connaissances en objets structurés ;
* **le DSL**, qui offre un langage pour décrire ces connaissances sans dépendre du code.

À partir du prochain chapitre, nous allons entrer dans une dimension beaucoup plus pragmatique. Nous analyserons **le système de plugins et d'extensions**, véritable moteur de l'évolutivité du framework. Nous verrons comment Searchlores sépare son cœur de ses capacités, comment il accueille de nouveaux outils d'analyse, de nouveaux moteurs de recherche ou de nouveaux formats de restitution, et pourquoi cette architecture est l'une de ses plus grandes forces sur le long terme.
