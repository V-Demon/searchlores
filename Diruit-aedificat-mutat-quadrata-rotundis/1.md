# Chapitre 1 — Searchlores : Réinventer la recherche comme discipline intellectuelle

> *« The Search is the Program. »*
> Cette phrase, omniprésente dans Searchlores, n'est pas un slogan marketing. C'est un changement profond de paradigme.

---

# Avant-propos

La plupart des frameworks destinés à l'intelligence artificielle commencent par une question simple :

> *Comment obtenir une meilleure réponse ?*

Searchlores, lui, pose une question radicalement différente :

> **Comment conduire une meilleure investigation ?**

Cette nuance paraît subtile.

En réalité, elle change absolument tout.

Là où LangChain, CrewAI, DSPy ou LangGraph considèrent qu'un LLM doit produire une réponse en orchestrant des appels d'API, Searchlores considère que la réponse n'est qu'un **artefact final**.

La véritable unité de travail devient...

> **l'enquête.**

---

# Une philosophie héritée de Fravia

Dès les premiers fichiers du dépôt, un nom revient régulièrement.

**Fravia.**

Pour comprendre Searchlores, il faut comprendre cet héritage.

Dans les années 1990, Fravia développait une approche baptisée **Searching**.

Il ne s'agissait pas de taper quelques mots-clés dans Altavista.

Il s'agissait d'apprendre à **penser comme un chercheur**.

Une recherche devenait :

* une exploration,
* une cartographie,
* une enquête,
* une reconstruction.

L'objectif n'était jamais de trouver une réponse.

L'objectif était de découvrir une structure.

Searchlores reprend presque intégralement cette philosophie.

---

# Le problème des IA modernes

Les LLM excellent pour produire du texte.

Mais ils présentent une faiblesse importante.

Ils répondent extrêmement vite.

Parfois…

beaucoup trop vite.

Un agent classique fonctionne souvent ainsi :

```text
Question
      │
      ▼
LLM
      │
      ▼
Réponse
```

Même lorsqu'on ajoute :

* des outils,
* du RAG,
* des appels Web,
* de la mémoire,

la logique reste essentiellement identique.

Le système cherche la **première réponse suffisamment bonne**.

Searchlores refuse cette logique.

---

# La réponse est un sous-produit

Dans Searchlores, la réponse finale n'est pas l'objectif.

Elle est le résultat d'une accumulation de connaissances.

Le processus ressemble davantage à ceci :

```text
Question
     │
     ▼
Observation
     │
     ▼
Hypothèses
     │
     ▼
Recherche
     │
     ▼
Découverte
     │
     ▼
Validation
     │
     ▼
Cartographie
     │
     ▼
Narration
     │
     ▼
Conclusion
```

La différence est fondamentale.

Le moteur n'essaie plus de répondre.

Il essaie de **comprendre.**

---

# Une inversion de paradigme

La plupart des frameworks IA organisent :

* des appels de modèles,
* des outils,
* des chaînes,
* des graphes d'exécution.

Searchlores organise...

des idées.

Cela paraît abstrait.

Pourtant, lorsqu'on parcourt le dépôt, on découvre rapidement que les objets principaux ne sont pas des "Tasks" ou des "Chains".

On rencontre des notions comme :

* Lore
* Investigation
* Narrative
* Archaeology
* Evidence
* Context
* Mapping

Le vocabulaire révèle immédiatement l'intention du projet.

Nous ne sommes plus dans un framework d'orchestration.

Nous sommes dans un framework **d'investigation cognitive**.

---

# Une IA qui enquête

Imaginez un historien.

Lorsqu'on lui pose une question, il ne répond pas immédiatement.

Il commence par :

* identifier les sources ;
* vérifier leur origine ;
* rechercher leurs contradictions ;
* établir une chronologie ;
* reconstruire un contexte ;
* seulement ensuite produire une synthèse.

Searchlores tente de reproduire ce comportement.

Le LLM cesse d'être une machine à répondre.

Il devient un enquêteur.

---

# Les quatre piliers de Searchlores

À la lecture du dépôt, quatre idées structurent l'ensemble du framework.

## 1. L'investigation

Tout part d'une investigation.

Une investigation possède :

* un sujet ;
* des hypothèses ;
* des pistes ;
* des preuves ;
* des résultats.

Autrement dit, Searchlores ne manipule pas des prompts.

Il manipule des enquêtes.

---

## 2. Le Lore

Le terme *Lore* est volontaire.

Dans les jeux vidéo, le lore désigne l'ensemble des connaissances qui donnent de la cohérence à un univers.

Searchlores reprend cette idée.

Le Lore représente la connaissance organisée.

Il ne s'agit pas simplement de documents.

Il s'agit :

* d'un contexte,
* d'une mémoire,
* d'un réseau de concepts.

Le framework considère que la connaissance est un objet vivant.

---

## 3. L'archéologie

C'est probablement la notion la plus originale.

Pourquoi parler d'archéologie ?

Parce qu'une information n'est jamais considérée comme évidente.

Elle possède une histoire.

Une origine.

Des couches.

Des transformations.

L'objectif devient alors de fouiller cette information comme un archéologue fouille un site antique.

Cette métaphore revient régulièrement dans le projet.

Elle influence directement l'architecture logicielle.

---

## 4. La narration

Une enquête produit des découvertes.

Encore faut-il pouvoir les raconter.

Le moteur distingue donc :

* les faits ;
* les preuves ;
* les relations ;
* le récit.

Cette séparation est extrêmement intéressante.

Elle rappelle les méthodes employées dans :

* le journalisme d'investigation ;
* l'intelligence économique ;
* la criminalistique ;
* la recherche académique.

---

# Une architecture orientée connaissance

L'un des premiers constats lorsque l'on explore le dépôt est que Searchlores ne cherche pas à encapsuler les LLM.

Il cherche à encapsuler la connaissance.

Autrement dit :

```
LLM
   ↓
est un outil

mais

Knowledge
   ↓
est le véritable cœur du système.
```

Cette distinction explique énormément de choix architecturaux que nous découvrirons dans les chapitres suivants.

---

# Pourquoi ce projet est différent

Comparons rapidement.

| Framework       | Objet principal   |
| --------------- | ----------------- |
| LangChain       | Chain             |
| LangGraph       | Graphe            |
| CrewAI          | Agent             |
| DSPy            | Programme         |
| Haystack        | Pipeline          |
| AutoGen         | Conversation      |
| **Searchlores** | **Investigation** |

Ce simple tableau résume presque toute la philosophie du projet.

---

# Ce que Searchlores n'essaie pas d'être

C'est un point essentiel.

Searchlores ne cherche pas à remplacer :

* un moteur RAG ;
* un orchestrateur d'agents ;
* un framework de prompts ;
* un SDK OpenAI.

Il cherche à fournir une couche supérieure.

Une couche où l'objet manipulé n'est plus le texte, mais la **connaissance structurée**.

---

# Une lecture du dépôt

À première vue, le dépôt peut sembler hétérogène.

On y trouve :

* du code Python ;
* des RFC ;
* des manifestes ;
* des spécifications ;
* des exemples ;
* des fichiers de configuration ;
* des plugins.

Ce n'est pas un hasard.

Le projet est encore en construction.

Il fonctionne autant comme un laboratoire d'idées que comme une bibliothèque logicielle.

Cette dualité est importante : certaines briques sont déjà opérationnelles, d'autres relèvent davantage de la feuille de route. Une bonne lecture consiste donc à distinguer les concepts stabilisés des expérimentations en cours.

---

# Ce que nous allons découvrir

Les chapitres suivants quitteront progressivement le terrain des idées pour entrer dans celui de l'implémentation.

Nous disséquerons :

* l'organisation complète du dépôt ;
* le rôle de chaque package Python ;
* le moteur d'investigation ;
* le système de plugins ;
* le DSL **Lore** ;
* les mécanismes d'extension ;
* les flux internes ;
* les cas d'utilisation réels.

Notre objectif ne sera pas simplement de comprendre **comment** Searchlores fonctionne.

Nous chercherons surtout à comprendre **pourquoi** il a été conçu ainsi.

Car c'est précisément dans cette intention que réside l'originalité du projet. Le code n'est pas seulement une implémentation : il matérialise une manière de penser la recherche, où l'enquête, le contexte et la construction progressive du savoir priment sur la génération immédiate d'une réponse. C'est cette perspective qui guidera toute la suite de cette monographie.
